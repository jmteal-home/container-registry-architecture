This document defines the complete observability architecture for the Enterprise Container Registry platform. It covers the OpenTelemetry instrumentation strategy, Log Analytics workspace design, Prometheus metrics, Grafana dashboards, WASM workload observability via wasmCloud OTEL integration, alert definitions, and the telemetry pipeline from each component to the centralized observability stack.

# 1. Observability Architecture Overview
The observability architecture follows the OpenTelemetry three-pillar model (logs, metrics, traces) for all custom components, supplemented by Azure Monitor native telemetry for Azure PaaS services. Every custom component — Token Broker, EMS integration adapter, entitlement cache — emits structured OTEL telemetry from day one.

| **Component** | **Logs** | **Metrics** | **Traces** | **Telemetry Destination** |
| --- | --- | --- | --- | --- |
| Azure Container Registry (ACR) | Diagnostic logs: ContainerRegistryLoginEvents, ContainerRegistryRepositoryEvents | Azure Monitor: SuccessfulPullCount, SuccessfulPushCount, StorageUsed, ReplicationLatency | N/A (Azure-managed) | Log Analytics workspace (audit); Azure Monitor metrics |
| Token Broker (custom) | Structured JSON logs: OTEL log appender → OTEL Collector → Log Analytics | Prometheus metrics: token_issuances_total, ems_latency_seconds, cache_hit_ratio (see [DOC-009 Section 7.2](../phase-3-entitlement/DOC-009_Token_Broker_Architecture.md)) | OTEL traces: token.issuance → jwt.validation → entitlement.cache.lookup → acr.token.generate | OTEL Collector → Azure Monitor / Log Analytics / Grafana |
| Entitlement System Integration | EMS API call logs: response time, status code, cache hit/miss | EMS API latency histogram, circuit breaker state gauge | ems.query span within token.issuance trace | Same as Token Broker |
| Azure Cache for Redis | Azure diagnostic logs: slow commands, errors | Azure Monitor: CacheHits, CacheMisses, ConnectedClients, UsedMemory | N/A | Log Analytics workspace |
| Azure Front Door (WAF) | WAF access logs: source IP, rule matched, blocked requests | Request count, latency, error rate by origin | N/A | Log Analytics workspace |
| wasmCloud / Cosmonic Control | OTEL logs: component lifecycle, capability invocations, error events | OTEL metrics: component_invocations_total, invocation_latency_seconds, active_components | OTEL distributed traces: across component boundaries via wRPC propagation | Customer-configured OTEL endpoint (Splunk, Datadog, Honeycomb, or company-provided) |
| CI/CD Pipelines | Pipeline build logs in ADO/GitHub | Push success/failure rate, scan pass rate, build duration | OTEL trace: build → scan → push → sign → sbom (generated by pipeline instrumentation) | Azure DevOps / GitHub Actions native; OTEL export to platform Log Analytics |
| Connected Registry (edge) | Sync status logs, client access logs | Sync latency, sync success rate, storage used | N/A | Log Analytics (when connected); local storage (edge) |

# 2. OpenTelemetry Pipeline Design

## 2.1 OTEL Collector Architecture
An OpenTelemetry Collector is deployed in each Azure region as a central telemetry aggregation and routing hub. All custom components export to the regional OTEL Collector, which then fans out to multiple backends:

| **OTEL Collector Role** | **Configuration** | **Backends** |
| --- | --- | --- |
| Receiver | OTLP receiver on port 4317 (gRPC) and 4318 (HTTP). Accepts telemetry from Token Broker, pipeline instrumentation, and WASM runtimes. | N/A — ingest only |
| Processor — Batch | Batch processor: max_export_batch_size: 512, timeout: 5s. Reduces API call volume to backends. | N/A — processing only |
| Processor — Resource | Adds common attributes to all telemetry: service.namespace=registry-platform, deployment.environment, cloud.region, host.name | N/A |
| Processor — Filter | Security filter: removes any telemetry containing customer_id in raw form (hashed before export to comply with data minimization) | N/A |
| Exporter — Azure Monitor | Exports logs and metrics to Log Analytics workspace. Uses Azure Monitor OTEL exporter with Managed Identity authentication. | Log Analytics (audit workspace + operations workspace) |
| Exporter — Prometheus | Exposes Prometheus metrics endpoint for Grafana scraping | Grafana (Prometheus datasource) |
| Exporter — Tempo/Jaeger | Exports traces to distributed tracing backend | Grafana Tempo (traces) |


## 2.2 Token Broker OTEL Configuration
```
# Token Broker OpenTelemetry SDK configuration
# Environment variables injected into ACA container OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector.{vnet-internal}:4317 OTEL_EXPORTER_OTLP_PROTOCOL=grpc OTEL_SERVICE_NAME=token-broker OTEL_SERVICE_VERSION={version} OTEL_RESOURCE_ATTRIBUTES=deployment.environment=production,cloud.region=eastus2
# Trace sampling: 100% for token issuances (always trace customer interactions) OTEL_TRACES_SAMPLER=always_on
# Metrics: push to OTEL Collector every 30 seconds OTEL_METRIC_EXPORT_INTERVAL=30000
# Logs: structured JSON with OTEL log bridge OTEL_LOGS_EXPORTER=otlp
```


# 3. Log Analytics Workspace Design
| **Workspace** | **Purpose** | **Retention** | **Access Control** | **Immutability** |
| --- | --- | --- | --- | --- |
| law-registry-audit | Tamper-evident audit log: all ACR access events, token issuances, RBAC changes, admin actions. SOC 2 and ISO 27001 evidence store. | 12 months hot + 24 months archive (36 months total) | Separate from operations workspace. Registry admins have NO access. Security Operations read-only via PIM. | Azure Monitor immutable storage policy: logs cannot be deleted or modified |
| law-registry-operations | Operational telemetry: Token Broker metrics, EMS integration health, pipeline telemetry, general system logs. | 3 months hot + 9 months archive | Platform Engineering read/write; on-call engineers read | Not immutable — operational logs may be purged per retention policy |
| law-registry-security | Security-specific telemetry: WAF logs, Defender for Containers findings, vulnerability scan results, signing events | 6 months hot + 18 months archive | Security Operations team; CISO office | Read-only after 7 days (semi-immutable via policy) |

# 4. Grafana Dashboard Specifications
The following Grafana dashboards are provisioned as code (dashboard JSON in platform IaC repository) and automatically deployed with the observability stack:

| **Dashboard** | **Primary Audience** | **Key Panels** | **Data Sources** |
| --- | --- | --- | --- |
| Registry Operations Overview | Platform Engineering on-call | ACR pull/push rate, Token Broker RPS, EMS health, circuit breaker state, cache hit ratio, active customer tokens, anomaly alerts | Azure Monitor (ACR metrics) + Prometheus (Token Broker) |
| Customer Pull Analytics | Customer Success, Platform Engineering | Pull rate by customer, pull rate by product namespace, token issuance count, denied requests by reason, top-10 customers by pull volume | Log Analytics (ContainerRegistryRepositoryEvents + Token Broker audit logs) |
| Security & Compliance | Security Operations, CISO | Failed authentication attempts, unauthorized access denials, emergency revocations, image signing events, quarantine events, CVE detections | Log Analytics (security workspace) + Defender for Cloud |
| Supply Chain Health | Platform Engineering, Security | Image signing rate, SBOM coverage %, quarantine pass/fail rate, CVE age (days since detection), promotion gate throughput | Log Analytics (operations workspace) + Defender for Cloud |
| WASM Workload Observability | Platform Engineering, Product Engineering (WASM teams) | WASM component pull rate, component deployment success rate, wasmCloud host health, OTEL trace overview for WASM invocations | Prometheus (WASM OTEL) + Grafana Tempo (traces) |
| SLO Burn Rate Dashboard | Platform Engineering, Management | Registry pull availability SLO, Token Broker latency SLO, error budget consumption rate, burn rate alerts | Prometheus (recording rules for SLO metrics) |

# 5. Alert Definitions
The following alerts are configured in Azure Monitor and Grafana. Alerts are organized by severity (P0-P3) and route to the appropriate on-call channel:

| **Alert Name** | **Condition** | **Severity** | **Route** | **Response SLO** |
| --- | --- | --- | --- | --- |
| Registry Availability Critical | ACR pull success rate < 99% over 5 minutes | P0 | PagerDuty → Platform Engineering on-call + Management | Acknowledge within 15 minutes; resolve within 1 hour |
| Token Broker Down | Token Broker health probe returns 503 for > 2 consecutive checks (60 seconds) | P0 | PagerDuty → Platform Engineering on-call | Acknowledge within 5 minutes |
| Emergency Revocation Executed | Emergency revocation telemetry event fired (any customer) | P0 (security) | PagerDuty → Security Operations + CISO | Immediate security incident investigation |
| EMS Circuit Breaker Open | Token Broker EMS circuit breaker transitions to OPEN state | P1 | PagerDuty → Platform Engineering + EMS team | Acknowledge within 15 minutes; EMS team notified |
| Token Broker P99 Latency High | Token Broker token issuance P99 > 500ms over 10 minutes | P2 | Teams channel → Platform Engineering on-call | Investigate within 1 hour |
| Image CVE Critical in Production | Microsoft Defender detects Critical CVE in production-namespace image | P1 (security) | PagerDuty → Security Operations + product team | Security investigation within 1 hour |
| Cache Hit Ratio Low | Token Broker cache hit ratio < 0.80 over 15 minutes | P2 | Teams channel → Platform Engineering | Investigate within 1 business day |
| Registry Storage > 80% | ACR storage utilization > 80% of configured soft limit | P3 | Teams channel → Platform Engineering | Review lifecycle policies within 3 business days |
| Signing Key Rotation Overdue | kv-sign-ci-pipeline key age > 365 days | P2 | Email → Platform Engineering Lead + CISO | Key rotation within 2 weeks |

# 6. WASM Observability Integration
wasmCloud and Cosmonic Control emit comprehensive OpenTelemetry signals that integrate with the platform's observability stack. The integration is configured per-customer based on their OTEL endpoint preference:

- Cosmonic Control OTEL support: full OTEL logs, metrics, and traces via standard OTEL configuration (OTEL_EXPORTER_OTLP_ENDPOINT). Integrates with Splunk, Datadog, Honeycomb, or company-provided Grafana/Tempo stack.

- wasmCloud OSS: OTEL_EXPORTER_OTLP_ENDPOINT environment variable on wasmCloud hosts. Exports component invocation traces, host health metrics, lattice events.

- Component boundary tracing: wRPC (WIT-over-RPC) propagates OTEL trace context across WASM component boundaries, enabling distributed traces that span multiple components.

- Platform telemetry vs component telemetry: Cosmonic Control provides clean separation — platform-level telemetry (host health, scheduling) is separate from component-level telemetry (business logic invocations). This simplifies triage.
